Campaign stage: keyLocations

Based on this campaignConcept: {{campaignConcept}}

Return **ONLY** YAML (no surrounding text). Preferred shape:

keyLocations:
  - name: "..."
    type: "..."
    significance: "..."

worldMap:
  regions:
    - id: "r1"
      name: "English region or march name"
      terrain: "short tag (forest, hills, coast, …)"
      neighborIds: ["r2", "r3"]
    - id: "r2"
      name: "..."
      terrain: "..."
      neighborIds: ["r1"]
  siteLinks:
    - locationName: "exact same string as one keyLocations[].name"
      regionId: "r1"
  startingRegionId: "r1"

**worldMap (required in the same YAML document):** A small **graph** the server and DM AI use for travel and “where things are.” Rules:
- **regions:** 4–12 items. Each **id** is a short stable token (e.g. `r1`, `north_marches`). **neighborIds** lists **adjacent** regions (overland travel); treat edges as **two-way** even if you list them on one side only. **name** is an **English** fantasy region name (not a real-world city).
- **siteLinks:** For **each** key location you listed, include one row linking its **name** to the **regionId** where that site mainly sits.
- **startingRegionId:** One **id** from **regions** where the implied campaign geography is centered (borders / travel hub is fine).

Each keyLocations item must have:
- name (string) — **English** fantasy place name (evocative or invented toponym; not real mundane city names), even when **type** / **significance** follow the **session language policy** (system message); vary geography and metaphor across locations
- type (string) — short type label (settlement, ruin, temple, wilderness, …)
- significance (1-2 sentences; use `|` for multiline if needed)

A bare YAML sequence of key-location mappings at the root is **not** enough — **worldMap** must appear as a sibling mapping under the same root as **keyLocations** (or under a single root mapping that includes both).
